Почему docker exec в Git Bash мог показать "No such file or directory"

Кратко:
Git Bash (MSYS) на Windows иногда преобразует аргументы путей (например `/var/www/html/.env`) в Windows-пути (`C:/Program Files/Git/var/www/html/.env`) перед тем, как передать их в `docker` CLI. В результате команда вроде

    docker exec backend cat /var/www/html/.env

может получить не тот аргумент и попытаться читать файл на хосте, а не внутри контейнера.

Что сделано в этом репозитории:
- Добавлен скрипт-обёртка `scripts/docker-exec.sh`, который гарантированно запускает переданную команду внутри контейнера через `bash -lc`, предотвращая трансляцию путей MSYS.

Как пользоваться (в Git Bash):

1. Сделайте скрипт исполняемым (один раз):

    chmod +x scripts/docker-exec.sh

2. Выполните команду внутри контейнера через обёртку, например:

    ./scripts/docker-exec.sh backend "cat /var/www/html/.env"

Это эквивалентно выполнению в контейнере:

    docker exec -it backend bash -lc 'cat /var/www/html/.env'

и безопасно от трансляции путей Git Bash.

Альтернативный вариант (без скрипта): всегда заключайте команду в одинарные кавычки и используйте `bash -lc`:

    docker exec -it backend bash -lc 'cat /var/www/html/.env'

Что делать дальше:
- Если нужно — используйте обёртку для всех последующих диагностических команд (логов, проверок конфигурации и т.п.).
- Если хотите, могу автоматически добавить пару npm/Make/PS-скриптов для запуска часто используемых команд.
